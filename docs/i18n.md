# Internationalization (i18n)

This project uses [next-intl](https://next-intl.dev/) to handle internationalization within the Next.js App Router.

## Supported Locales

| Locale | Language | URL prefix         |
| ------ | -------- | ------------------ |
| `es`   | Spanish  | _(none — default)_ |
| `en`   | English  | `/en/`             |

The default locale is **Spanish (`es`)**. Because `localePrefix` is set to `"as-needed"`, the default locale is served without a prefix (e.g. `/`) while the non-default locale includes the prefix (e.g. `/en/`).

## File Structure

```
services/i18n/
  routing.ts      # Locale list, default locale, prefix strategy
  request.ts      # Per-request config: locale resolution + message loading
  navigation.ts   # Locale-aware navigation helpers

messages/
  es.json         # Spanish translations (default)
  en.json         # English translations

app/
  layout.tsx              # Root layout (no locale logic)
  [locale]/
    layout.tsx            # Locale layout: validates locale, mounts NextIntlClientProvider
    (homepage)/
      page.tsx            # Uses useTranslations() to render translated content
      components/
        contact-form/
          contact-form.tsx  # Uses useTranslations() in a client component
```

## How It Works

### 1. Routing — `services/i18n/routing.ts`

Defines the central routing configuration consumed by all other i18n modules:

```ts
export const routing = defineRouting({
  locales: ["es", "en"],
  defaultLocale: "es",
  localePrefix: "as-needed",
});
```

### 2. Request Config — `services/i18n/request.ts`

Runs on every request (server-side). It resolves the active locale and dynamically loads the corresponding message file:

```ts
export default getRequestConfig(async ({ requestLocale }) => {
  const requested = await requestLocale;
  const locale = hasLocale(routing.locales, requested)
    ? requested
    : routing.defaultLocale;

  return {
    locale,
    messages: (await import(`@/messages/${locale}.json`)).default,
  };
});
```

- If the requested locale is not in the supported list, it falls back to the default locale (`es`).
- Messages are loaded lazily via a dynamic `import()`, so only the active locale's file is bundled per request.

This file is registered in `next.config.js` via the next-intl plugin:

```js
const withNextIntl = createNextIntlPlugin("./services/i18n/request.ts");
```

### 3. Navigation Helpers — `services/i18n/navigation.ts`

Exports locale-aware wrappers around Next.js navigation APIs:

```ts
export const { Link, redirect, usePathname, useRouter, getPathname } =
  createNavigation(routing);
```

Use these instead of the native Next.js equivalents to ensure locale information is automatically included in URLs.

### 4. Locale Layout — `app/[locale]/layout.tsx`

The `[locale]` dynamic segment captures the locale from the URL. The layout:

1. Validates the locale against the supported list — calls `notFound()` for unknown locales.
2. Sets the `lang` attribute on `<html>`.
3. Wraps children in `<NextIntlClientProvider>` to make translations available to Client Components.
4. Uses `getTranslations` to generate locale-aware page metadata.

```tsx
if (!hasLocale(routing.locales, locale)) {
  notFound();
}

return (
  <html lang={locale}>
    <body>
      <NextIntlClientProvider>{children}</NextIntlClientProvider>
    </body>
  </html>
);
```

## Using Translations in Components

### Server Components

Use the async `getTranslations` from `next-intl/server`:

```tsx
import { getTranslations } from "next-intl/server";

// Inside an async Server Component or generateMetadata
const t = await getTranslations({ locale, namespace: "homepage" });
t("metadata.title"); // → "Mateo Olarte - Software Engineer"
```

### Client Components (and Server Components via hook)

Use the synchronous `useTranslations` hook:

```tsx
import { useTranslations } from "next-intl";

function MyComponent() {
  const t = useTranslations();
  return <p>{t("homepage.title")}</p>;
}
```

A namespace can be passed to `useTranslations("homepage")` to avoid repeating the namespace prefix on every key access.

## Message Files

Translation files live in `messages/` and are structured as nested JSON objects. The top-level keys act as **namespaces**:

```json
{
  "common": {
    "jobTitle": "Software Engineer"
  },
  "homepage": {
    "metadata": {
      "title": "Mateo Olarte - Software Engineer",
      "description": "..."
    },
    "title": "New website coming soon",
    ...
  }
}
```

Keys are accessed with dot notation: `t("homepage.metadata.title")` or, when a namespace is set, just `t("metadata.title")`.

## Adding a New Locale

1. Add the locale code to the `locales` array in `services/i18n/routing.ts`.
2. Create the corresponding message file at `messages/<locale>.json` with all required keys.
3. No further configuration is needed — the request config and navigation helpers pick up the new locale automatically.

## Adding a New Translation Key

1. Add the key to **all** message files (`es.json`, `en.json`, etc.) to avoid missing-translation warnings at runtime.
2. Reference the key in components via `t("namespace.key")`.
